    Core Concepts (25 Questions)

    241. What is MySQL and why is it popular?

    MySQL is an open-source relational database management system (RDBMS) that uses Structured Query Language (SQL). It's popular because:

    - Open Source: Free to use with strong community support

    - Performance: Fast and reliable for web applications

    - Scalability: Handles large datasets efficiently

    - Cross-platform: Runs on various operating systems

    - ACID Compliance: Ensures data integrity

    - Wide Adoption: Used by major companies like Facebook, Twitter, YouTube

    242. What is a Relational Database Management System (RDBMS)?

    An RDBMS is a database management system based on the relational model where:

    - Data is stored in tables (relations)

    - Tables consist of rows (tuples) and columns (attributes)

    - Relationships between tables are established through keys

    - Follows ACID properties (Atomicity, Consistency, Isolation, Durability)

    - Uses SQL for data manipulation and querying

    243. What is a database schema?

    A database schema is the logical structure that defines:

    - Tables: What tables exist and their structure

    - Columns: Data types, constraints, and properties

    - Relationships: How tables relate to each other

    - Indexes: Performance optimization structures

    - Views: Virtual tables based on queries

    - Stored Procedures: Reusable code blocks

    Example: An e-commerce schema might include tables for users, products, orders, and order_items.

    244. What are tables, rows, and columns?

    - Table: A collection of related data organized in rows and columns (like a spreadsheet)

    - Row (Record/Tuple): A single instance of data representing one entity

    - Column (Field/Attribute): A specific data element that all rows share

    Example:

    plaintext 
    Users Table:
| id | name    | email           | age |
|----|---------|-----------------|-----|
| 1  | John    | john@email.com  | 25  |
| 2  | Sarah   | sarah@email.com | 30  |

    245. What are the main SQL data types in MySQL?

    Numeric Types:

    - `INT`: Integer numbers (-2,147,483,648 to 2,147,483,647)

    - `BIGINT`: Large integers

    - `DECIMAL(M,D)`: Fixed-point numbers

    - `FLOAT`: Single-precision floating-point

    - `DOUBLE`: Double-precision floating-point

    String Types:

    - `VARCHAR(n)`: Variable-length strings (up to n characters)

    - `CHAR(n)`: Fixed-length strings

    - `TEXT`: Large text data

    - `LONGTEXT`: Very large text data

    Date/Time Types:

    - `DATE`: Date values (YYYY-MM-DD)

    - `TIME`: Time values (HH:MM:SS)

    - `DATETIME`: Date and time

    - `TIMESTAMP`: Timestamp with timezone

    246. What is a Primary Key and what are its properties?

    A Primary Key is a column (or combination of columns) that uniquely identifies each row in a table.

    Properties:

    - Uniqueness: No two rows can have the same primary key value

    - Non-null: Cannot contain NULL values

    - Immutable: Should not change once assigned

    - Minimal: Should use the fewest columns necessary

    Example: In a users table, `user_id` serves as the primary key.

    247. What is a Foreign Key and its purpose?

    A Foreign Key is a column that creates a link between two tables by referencing the primary key of another table.

    Purpose:

    - Referential Integrity: Ensures data consistency across tables

    - Relationships: Establishes connections between related data

    - Constraints: Prevents invalid data entry

    Example: In an orders table, `customer_id` is a foreign key referencing the `id` in the customers table.

    248. What is a Composite Key?

    A Composite Key is a primary key consisting of multiple columns that together uniquely identify a row.

    Example: In an order_items table:

    - `order_id` + `product_id` forms a composite key

    - Neither column alone is unique, but together they are

    - Useful for many-to-many relationship tables

    249. What is a Unique constraint?

    A Unique constraint ensures that all values in a column (or combination of columns) are different from each other.

    Differences from Primary Key:

    - Can have multiple unique constraints per table

    - Can contain NULL values (but only one NULL per column)

    - Not automatically indexed (though often are)

    Example: Email addresses in a users table should be unique.

    250. What is a Default constraint?

    A Default constraint provides a default value for a column when no value is specified during insertion.

    Examples:

    - `created_at DATETIME DEFAULT CURRENT_TIMESTAMP`

    - `status VARCHAR(20) DEFAULT 'active'`

    - `is_verified BOOLEAN DEFAULT FALSE`

    251. What is a NULL value and how is it different from empty string?

    NULL: Represents the absence of any value or unknown value

    Empty String (''): A string with zero length but still a value

    Key Differences:

    - NULL takes no storage space; empty string takes minimal space

    - NULL comparisons use `IS NULL`/`IS NOT NULL`; empty string uses `= ''`

    - NULL in calculations results in NULL; empty string participates normally

    252. What are the CRUD operations and their SQL commands?

    CRUD stands for Create, Read, Update, Delete:

    - Create: `INSERT INTO table_name (columns) VALUES (values)`

    - Read: `SELECT columns FROM table_name WHERE condition`

    - Update: `UPDATE table_name SET column = value WHERE condition`

    - Delete: `DELETE FROM table_name WHERE condition`

    253. What is the WHERE clause and what operators can be used?

    The WHERE clause filters records based on specified conditions.

    Operators:

    - Comparison: `=`, `!=`, `<>`, `<`, `>`, `<=`, `>=`

    - Logical: `AND`, `OR`, `NOT`

    - Pattern: `LIKE`, `NOT LIKE`

    - Range: `BETWEEN`, `NOT BETWEEN`

    - List: `IN`, `NOT IN`

    - Null: `IS NULL`, `IS NOT NULL`

    254. What is the ORDER BY clause?

    ORDER BY sorts the result set by one or more columns.

    Syntax: `ORDER BY column1 ASC/DESC, column2 ASC/DESC`

    - ASC: Ascending order (default)

    - DESC: Descending order

    - Can sort by multiple columns with different directions

    255. What is the DISTINCT keyword?

    DISTINCT removes duplicate rows from the result set, returning only unique values.

    Example: `SELECT DISTINCT city FROM users` returns each city only once, even if multiple users live there.

    256. What is the LIMIT clause?

    LIMIT restricts the number of rows returned by a query.

    Syntax:

    - `LIMIT count`: Returns first 'count' rows

    - `LIMIT offset, count`: Skips 'offset' rows, returns next 'count' rows

    Example: `SELECT * FROM users LIMIT 10` returns first 10 users.

    257. What is the LIKE operator and wildcards?

    LIKE is used for pattern matching in string comparisons.

    Wildcards:

    - `%`: Matches zero or more characters

    - `_`: Matches exactly one character

    Examples:

    - `name LIKE 'John%'`: Names starting with "John"

    - `email LIKE '%@gmail.com'`: Gmail addresses

    - `phone LIKE '555-____'`: Phone numbers starting with 555

    258. What are Aggregate Functions?

    Aggregate functions perform calculations on multiple rows and return a single result.

    Common Functions:

    - `COUNT()`: Counts rows

    - `SUM()`: Adds numeric values

    - `AVG()`: Calculates average

    - `MIN()`: Finds minimum value

    - `MAX()`: Finds maximum value

    259. What is the GROUP BY clause?

    GROUP BY groups rows with the same values in specified columns and allows aggregate functions to be applied to each group.

    Example: `SELECT city, COUNT(*) FROM users GROUP BY city` counts users per city.

    260. What is the HAVING clause and how is it different from WHERE?

    HAVING filters groups created by GROUP BY, while WHERE filters individual rows.

    Key Differences:

    - WHERE: Filters rows before grouping

    - HAVING: Filters groups after grouping

    - HAVING: Can use aggregate functions in conditions

    - WHERE: Cannot use aggregate functions

    261. What are Scalar Functions?

    Scalar functions operate on individual values and return a single value for each row.

    Examples:

    - String: `UPPER()`, `LOWER()`, `LENGTH()`, `SUBSTRING()`

    - Numeric: `ROUND()`, `ABS()`, `CEIL()`, `FLOOR()`

    - Date: `NOW()`, `YEAR()`, `MONTH()`, `DATE_FORMAT()`

    262. What is the difference between CHAR and VARCHAR?

    CHAR(n):

    - Fixed-length string

    - Always uses n bytes of storage

    - Faster for fixed-size data

    - Padded with spaces if shorter

    VARCHAR(n):

    - Variable-length string

    - Uses only needed storage + 1-2 bytes for length

    - More storage efficient

    - No padding

    263. What is the difference between DATETIME and TIMESTAMP?

    DATETIME:

    - Range: 1000-01-01 to 9999-12-31

    - 8 bytes storage

    - No timezone awareness

    - Not affected by server timezone changes

    TIMESTAMP:

    - Range: 1970-01-01 to 2038-01-19

    - 4 bytes storage

    - Timezone aware

    - Automatically converts based on server timezone

    264. What are MySQL storage engines?

    Storage engines determine how MySQL stores, retrieves, and manages data.

    Popular Engines:

    - InnoDB: Default, supports transactions, foreign keys

    - MyISAM: Fast reads, no transactions

    - Memory: Stores data in RAM

    - Archive: For compressed, archived data

    265. What is the difference between InnoDB and MyISAM?

    InnoDB:

    - ACID compliant with transactions

    - Row-level locking

    - Foreign key support

    - Crash recovery

    - Better for write-heavy applications

    MyISAM:

    - No transaction support

    - Table-level locking

    - No foreign keys

    - Faster for read-heavy applications

    - Smaller storage footprint

    Joins & Relationships (30 Questions)

    291. What is a SQL Join and why is it necessary?

    A SQL Join combines rows from two or more tables based on a related column between them. It's necessary because:

    - Normalization: Data is split across multiple tables to avoid redundancy

    - Relationships: Real-world data has relationships that need to be queried together

    - Efficiency: Avoids data duplication and maintains data integrity

    - Flexibility: Allows complex queries across multiple entities

    292. Explain INNER JOIN with example.

    INNER JOIN returns only rows that have matching values in both tables.

    sql 
    SELECT u.name, o.order_date, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

    This returns only users who have placed orders, along with their order details.

    293. Explain LEFT (OUTER) JOIN with example.

    LEFT JOIN returns all rows from the left table and matching rows from the right table. If no match, NULL values for right table columns.

    sql 
    SELECT u.name, o.order_date, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

    This returns all users, including those who haven't placed any orders (with NULL order details).

    294. Explain RIGHT (OUTER) JOIN with example.

    RIGHT JOIN returns all rows from the right table and matching rows from the left table.

    sql 
    SELECT u.name, o.order_date, o.total
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id;

    This returns all orders, including those without valid user references.

    295. Explain FULL (OUTER) JOIN (how to simulate in MySQL).

    FULL OUTER JOIN returns all rows when there's a match in either table. MySQL doesn't support FULL OUTER JOIN directly, but can be simulated:

    sql 
    SELECT u.name, o.order_date, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
UNION
SELECT u.name, o.order_date, o.total
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id;

    296. What is a CROSS JOIN?

    CROSS JOIN returns the Cartesian product of both tables - every row from the first table combined with every row from the second table.

    sql 
    SELECT u.name, p.product_name
FROM users u
CROSS JOIN products p;

    If users has 10 rows and products has 5 rows, result will have 50 rows.

    297. What is a Self Join?

    A Self Join joins a table with itself, useful for hierarchical data or comparing rows within the same table.

    sql 
    SELECT e1.name as employee, e2.name as manager
FROM employees e1
LEFT JOIN employees e2 ON e1.manager_id = e2.id;

    298. What is the difference between ON and USING clauses in joins?

    ON Clause:

    - More flexible, allows any condition

    - Can compare different column names

    - Can include additional conditions

    USING Clause:

    - Simpler syntax for columns with same names

    - Automatically eliminates duplicate columns

    - Only works when join columns have identical names

    sql 
    -- ON clause
SELECT * FROM users u JOIN orders o ON u.id = o.user_id;

-- USING clause (if both columns are named 'id')
SELECT * FROM users JOIN orders USING(id);

    311. How do you implement many-to-many relationships?

    Many-to-many relationships require a junction (bridge) table containing foreign keys from both related tables.

    Example: Students and Courses

    - Students table: student_id (PK), name, email

    - Courses table: course_id (PK), course_name, credits

    - Student_Courses table: student_id (FK), course_id (FK), enrollment_date

    312. How do you handle hierarchical data in MySQL?

    Adjacency List Model:

    - Each record has a parent_id pointing to its parent

    - Simple but requires recursive queries for deep hierarchies

    Nested Set Model:

    - Each node has left and right values

    - More complex to maintain but efficient for queries

    Path Enumeration:

    - Store the full path as a string

    - Easy to query ancestors and descendants

    314. How do you optimize JOIN operations?

    Optimization Strategies:

    - Indexes: Create indexes on join columns

    - Query Order: Put most selective table first

    - LIMIT Early: Use LIMIT when possible

    - Avoid Functions: Don't use functions on join columns

    - Proper Data Types: Ensure join columns have same data types

    - Statistics: Keep table statistics updated

    315. What is the difference between EXISTS and IN clauses?

    EXISTS:

    - Returns TRUE/FALSE based on subquery results

    - Stops at first match (more efficient)

    - Handles NULL values better

    - Better for large datasets

    IN:

    - Compares values against a list

    - Evaluates all values in the list

    - Can have issues with NULL values

    - Better for small, static lists
