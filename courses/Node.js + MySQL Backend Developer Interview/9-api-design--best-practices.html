    REST API Design (20 Questions)

    501. What is REST and what are its architectural constraints?

    REST (Representational State Transfer) is an architectural style for designing networked applications. It has six key constraints:

    1. Client-Server: Separation of concerns between client and server

    2. Stateless: Each request contains all information needed to process it

    3. Cacheable: Responses must define themselves as cacheable or non-cacheable

    4. Uniform Interface: Consistent way to interact with resources

    5. Layered System: Architecture can be composed of hierarchical layers

    6. Code on Demand (optional): Server can send executable code to client

    Example: A REST API for a blog would have endpoints like `/api/posts` (collection) and `/api/posts/123` (specific resource).

    502. What are Resources and URIs in REST?

    Resources are the key abstraction in REST - anything that can be named and addressed (users, posts, orders, etc.). URIs (Uniform Resource Identifiers) are the addresses used to identify these resources.

    Examples:

    - Resource: User â†’ URI: `/api/users/123`

    - Resource: User's posts â†’ URI: `/api/users/123/posts`

    - Resource: Specific post â†’ URI: `/api/posts/456`

    503. What are the HTTP methods and their properties?

    - GET: Retrieve data (Safe, Idempotent, Cacheable)

    - POST: Create new resources (Not safe, Not idempotent)

    - PUT: Update/replace entire resource (Not safe, Idempotent)

    - PATCH: Partial update (Not safe, Not idempotent)

    - DELETE: Remove resource (Not safe, Idempotent)

    - HEAD: Get headers only (Safe, Idempotent, Cacheable)

    - OPTIONS: Get allowed methods (Safe, Idempotent)

    504. What are the key principles of RESTful API design?

    1. Resource-based URLs: Use nouns, not verbs (`/users` not `/getUsers`)

    2. HTTP methods for actions: Use appropriate HTTP verbs

    3. Stateless communication: No server-side session state

    4. Consistent naming: Use plural nouns for collections

    5. Proper status codes: Return meaningful HTTP status codes

    6. JSON format: Use JSON for data exchange

    7. Versioning: Plan for API evolution

    505. What is HATEOAS?

    HATEOAS (Hypermedia as the Engine of Application State) means the API response includes links to related actions/resources, making the API self-discoverable.

    Example:

    json 
    {
  "id": 123,
  "name": "John Doe",
  "links": {
    "self": "/api/users/123",
    "posts": "/api/users/123/posts",
    "edit": "/api/users/123",
    "delete": "/api/users/123"
  }
}

    506. How do you version your APIs?

    Common versioning strategies:

    1. URL versioning: `/api/v1/users`, `/api/v2/users`

    2. Header versioning: `Accept: application/vnd.api+json;version=1`

    3. Query parameter: `/api/users?version=1`

    4. Content negotiation: `Accept: application/vnd.myapi.v1+json`

    Best practice: URL versioning is most common and clear.

    507. What are the common HTTP status codes?

    - 2xx Success: 200 (OK), 201 (Created), 204 (No Content)

    - 3xx Redirection: 301 (Moved Permanently), 304 (Not Modified)

    - 4xx Client Error: 400 (Bad Request), 401 (Unauthorized), 403 (Forbidden), 404 (Not Found), 409 (Conflict)

    - 5xx Server Error: 500 (Internal Server Error), 502 (Bad Gateway), 503 (Service Unavailable)

    508. How do you choose the right status code?

    Guidelines:

    - 200: Successful GET, PUT, PATCH

    - 201: Successful POST (resource created)

    - 204: Successful DELETE or PUT with no response body

    - 400: Invalid request data/format

    - 401: Authentication required

    - 403: Authenticated but not authorized

    - 404: Resource not found

    - 409: Conflict (duplicate resource)

    - 422: Validation errors

    - 500: Server-side errors

    509. How do you implement API pagination?

    Common approaches:

    1. Offset-based: `?page=2&limit=10` or `?offset=20&limit=10`

    2. Cursor-based: `?cursor=abc123&limit=10`

    3. Page-based: `?page=2&per_page=10`

    Response format:

    json 
    {
  "data": [...],
  "pagination": {
    "current_page": 2,
    "total_pages": 10,
    "total_items": 100,
    "per_page": 10,
    "next_page": "/api/users?page=3&limit=10",
    "prev_page": "/api/users?page=1&limit=10"
  }
}

    510. How do you implement filtering, sorting, and searching?

    Filtering: `?status=active&role=admin`

    Sorting: `?sort=created_at&order=desc` or `?sort=-created_at`

    Searching: `?search=john` or `?q=john`

    Combined: `?status=active&sort=name&order=asc&search=john&page=1&limit=10`

    511. How do you handle bulk operations in REST APIs?

    Approaches:

    1. Batch endpoint: `POST /api/users/batch` with array of operations

    2. Collection operations: `PATCH /api/users` with filter criteria

    3. Async processing: Return job ID for long-running operations

    Example:

    json 
    POST /api/users/batch
{
  "operations": [
    {"action": "create", "data": {...}},
    {"action": "update", "id": 123, "data": {...}},
    {"action": "delete", "id": 456}
  ]

    512. How do you implement partial updates (PATCH)?

    PATCH should only update provided fields:

    Request: `PATCH /api/users/123`

    json 
    {
  "email": "newemail@example.com",
  "status": "active"
}

    Only `email` and `status` fields are updated, other fields remain unchanged.

    513. How do you handle nested resources?

    Design patterns:

    1. Nested URLs: `/api/users/123/posts` (posts belonging to user 123)

    2. Flat structure with filters: `/api/posts?user_id=123`

    3. Embedded resources: Include related data in response

    Guidelines:

    - Limit nesting to 2-3 levels maximum

    - Use flat structure for complex relationships

    - Consider performance implications

    Advanced API Concepts (20 Questions)

    521. What is GraphQL and how does it compare to REST?

    GraphQL is a query language and runtime for APIs that allows clients to request exactly the data they need.

    Comparison:

    - Data fetching: GraphQL - single request for multiple resources; REST - multiple requests

    - Over/under-fetching: GraphQL - precise data; REST - fixed response structure

    - Versioning: GraphQL - schema evolution; REST - URL versioning

    - Caching: REST - better HTTP caching; GraphQL - more complex caching

    - Learning curve: REST - simpler; GraphQL - steeper learning curve

    522. When would you choose GraphQL over REST?

    Choose GraphQL when:

    - Mobile applications: Bandwidth optimization needed

    - Complex data relationships: Many interconnected resources

    - Rapid frontend development: Frontend teams need flexibility

    - Multiple clients: Different data requirements per client

    - Real-time features: Subscriptions needed

    Choose REST when:

    - Simple CRUD operations: Straightforward data operations

    - Caching important: Heavy reliance on HTTP caching

    - File uploads: Binary data handling

    - Team expertise: Team familiar with REST patterns

    523. How do you implement real-time APIs?

    Technologies:

    1. WebSockets: Full-duplex communication

    2. Server-Sent Events (SSE): Server-to-client streaming

    3. Long polling: Extended HTTP requests

    4. WebRTC: Peer-to-peer communication

    Use cases: Chat applications, live updates, notifications, collaborative editing

    524. What are WebSockets and when do you use them?

    WebSockets provide persistent, full-duplex communication between client and server.

    Characteristics:

    - Persistent connection: No request/response cycle

    - Low latency: Real-time communication

    - Bidirectional: Both client and server can initiate communication

    - Protocol upgrade: Starts as HTTP, upgrades to WebSocket

    Use cases: Gaming, chat, live trading, collaborative tools, real-time dashboards

    525. What are Server-Sent Events (SSE)?

    SSE enables server to push data to client over a single HTTP connection.

    Characteristics:

    - Unidirectional: Server to client only

    - HTTP-based: Uses standard HTTP

    - Automatic reconnection: Built-in reconnection logic

    - Event streaming: Structured event format

    Use cases: Live feeds, notifications, progress updates, monitoring dashboards

    526. How do you implement webhooks?

    Webhooks are HTTP callbacks triggered by events.

    Implementation steps:

    1. Event detection: Monitor for specific events

    2. Payload creation: Format event data

    3. HTTP request: POST to registered URL

    4. Retry logic: Handle failed deliveries

    5. Security: Sign requests, verify endpoints

    Example flow: User registers â†’ Event occurs â†’ Webhook fired â†’ External system notified

    534. How do you handle API circuit breakers?

    Circuit breaker pattern prevents cascading failures:

    States:

    1. Closed: Normal operation

    2. Open: Failing fast, blocking requests

    3. Half-open: Testing if service recovered

    Benefits: Prevents system overload, faster failure detection, graceful degradation

    535. How do you implement API retries?

    Retry strategies:

    1. Exponential backoff: Increasing delays between retries

    2. Fixed delay: Constant delay between retries

    3. Jittered backoff: Random component to prevent thundering herd

    4. Circuit breaker integration: Stop retries when circuit opens

    Considerations: Idempotent operations only, maximum retry limits, timeout handling
